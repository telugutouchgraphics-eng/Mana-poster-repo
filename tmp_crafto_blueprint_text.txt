Rendering, export algorithms, performance, security, roles
Template schema & layout engine
Server-driven template JSON (recommended): - canvas: { width, height, safeMargins } -
layers[]: each layer has: -  type: backgroundImage / text / imagePlaceholder / sticker / shape -
rect: { x, y, w, h } in normalized coords (0..1) - style: fontFamilyRef, fontSize, color , shadow,
align - bindings: e.g., "text": "quote.text", "image": "profile.photo"
Runtime binding: - Load template JSON - Pull user profile - Resolve bindings → build render tree
Text fit algorithm (core UX) - Input: bounding box, max lines, target font, text - Binary search font size
(high→low)  until  TextPainter fits  -  For  scripts  like  తెలగ/హిందీ,  ensure  proper  font  fallback
(download font assets per language).
Export pipeline (Flutter-accurate)
Flutter official API suggests: - Wrap the “poster canvas widget” with  RepaintBoundary (separate
display list; can improve repaint performance). 
- Capture via  RenderRepaintBoundary.toImage(pixelRatio: …) once paint phase completed.
- Convert ui.Image → bytes via toByteData(format: ImageByteFormat.png); PNG is lossless
and suited for text edges. 
Practical guardrails (performance): - toImage() returns raw RGBA in memory (large canvases can
spike RAM); pixelRatio should be capped based on device tier . 
- UI jank avoidance: heavy work should not block frames; Flutter performance guidance targets 60fps
(≈16ms/frame). 
- Cache expensive assets (fonts/backgrounds) and prefetch thumbnails.
Compression & file size strategy
Default export PNG for crisp text; allow JPEG for “photo-heavy” posters when user wants smaller
file.
Optional “server-side optimize” endpoint:
User uploads PNG
Server generates WEBP/JPEG variants for sharing targets
Store in object storage + CDN
If you do client-side manipulation, Dart  image library supports load/manipulate/save for multiple
formats. 
Scalability considerations
Template delivery: thumbnails/previews served via CDN; template JSON cacheable (ETag; 1–7
days TTL).
Search: precomputed indexes; language-aware tokenization.
Peak traffic (festivals): pre-warm CDN + rate limit OTP/billing endpoints.
Offline-first: cache last N templates per category; allow editing exports offline.

Security & privacy checklist
Baseline  controls  (aligned  with  OWASP  mobile  risks):  -  Secure  auth/session  storage,  least-privilege,
secure comms, input validation, secure local storage, supply-chain hygiene. 
- Storage access rules: if using Firebase Storage, Security Rules integrate with Firebase Auth and can
restrict path-based access + size/type limits. 
- Data minimisation: store only what’s needed; implement delete-account workflow (Play Store signals
user can request deletion; Mana Poster should match). 
- Consent + transparency: align with Apple privacy label expectations (device ID, diagnostics, photos/
videos). 
Admin, creator, manager roles & permissions
Role design (recommended): -  Admin: everything, including billing configs, bans, system settings. -
Manager: catalog ops (categories, featuring, language packs), analytics dashboards, promo banners. -
Creator: create templates/quotes, upload assets; cannot approve self-published content. - Moderator:
handle reports, takedowns, suspensions; sees moderation queue + audit logs.
This structure maps well to Crafto governance emphasis (moderation + takedown).
